Migrating a Production Platform from React to Angular: Lessons Learned

What I learned from leading a large-scale React to Angular migration in a live production environment — incremental strategy, team buy-in, and the Kendo UI quirks nobody warned me about.

reactangularmigrationtypescriptkendo-ui

When you're handed a live production React codebase and told to migrate it to Angular incrementally — without downtime, without breaking editorial workflows, and while still shipping features — you find out quickly which migration strategies actually hold up.

This is what I learned doing exactly that at Sports Business Journal.

Why Migrate in the First Place?

The short answer: standardization. The org was going all-in on Angular for new products, and having two frontend stacks meant duplicating tooling, training, and architectural decisions. React was working fine, but the long-term maintenance cost of maintaining two ecosystems wasn't worth it.

The harder question is how you do it without blowing up a system that real editors depend on every day.

The Incremental Strategy

We didn't do a big-bang rewrite. Instead, we ran React and Angular side-by-side using Angular Elements — Angular components compiled to native custom elements that can drop into any HTML page, including a React app.

The rough phases looked like this:

  1. Audit and slice — Identify self-contained UI sections (sidebars, widgets, data tables) that have clean props-in/events-out boundaries.
  2. Build new features in Angular only — Any greenfield work goes Angular. No new React components.
  3. Migrate leaf components first — Start with things that have no children, then work inward.
  4. Replace pages one at a time — Once enough components are migrated, flip entire route segments.

This let us ship value continuously instead of freezing the codebase for months.

Kendo UI: More Powerful Than Advertised, More Opinionated Too

The Kendo UI for Angular library was a requirement for this project, and it's genuinely good for data-heavy interfaces — grids, date pickers, dropdowns with virtualization. But it has a learning curve around change detection.

The biggest gotcha: Kendo components use ChangeDetectionStrategy.OnPush internally, which means you need to be disciplined about immutable data. Mutating an array and expecting the grid to re-render won't work. You have to spread or reconstruct your collections.

// This won't trigger re-render
this.rows.push(newRow);

// This will
this.rows = [...this.rows, newRow];

Once that clicked, the components became much more predictable.

What I'd Do Differently

  • Start the component library earlier. We built shared UI primitives too late, which led to drift between migrated and un-migrated sections.
  • Invest in E2E tests before touching core pages. We had good unit coverage but not enough integration tests, which made some page-level migrations risky.
  • Document the migration contract explicitly. An informal "leaf-first" rule means different things to different people. Write it down.

Where We Landed

The migration is ongoing, but the majority of core editorial and consumer-facing pages are now Angular. Performance improved noticeably — Angular's stricter change detection caught a lot of unnecessary re-renders that React's reconciler was quietly letting through.

If you're facing a similar migration, the most important thing is patience with the seams. The interfaces between the old and new system are where bugs live. Keep them small, test them hard, and eliminate them as quickly as the team can manage.
