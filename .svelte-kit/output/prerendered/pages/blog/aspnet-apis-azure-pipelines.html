Building Robust ASP.NET Core Web APIs and Deploying with Azure Pipelines

A practical pattern for structuring ASP.NET Core Web APIs for maintainability and high availability, then wiring up a zero-touch Azure DevOps deployment pipeline that actually stays green.

aspnetcsharpazuredevopspipelinesapirest

A Web API that works on your laptop but flakes under real load, or a deployment process that requires babysitting every release — both kill developer confidence fast. Here's the approach I've settled on for building ASP.NET Core APIs that hold up in production, paired with an Azure DevOps pipeline that handles releases reliably.

Project Structure

Flat controller files don't scale. For anything beyond a few endpoints, I separate concerns into layers:

src/
  Api/           ← Controllers, middleware, DI setup
  Application/   ← Business logic, command/query handlers (MediatR)
  Domain/        ← Entities, value objects, domain events
  Infrastructure/← EF Core, repositories, external service clients

This isn't CQRS dogma — it's just enough separation so that when a controller needs to change, it doesn't drag business logic with it.

Controllers Stay Thin

Controllers should do three things: parse the request, call a handler, return a response. That's it.

[ApiController]
[Route("api/[controller]")]
public class ArticlesController : ControllerBase
{
    private readonly IMediator _mediator;

    public ArticlesController(IMediator mediator) => _mediator = mediator;

    [HttpGet("{id:guid}")]
    public async Task<IActionResult> GetById(Guid id)
    {
        var result = await _mediator.Send(new GetArticleQuery(id));
        return result is null ? NotFound() : Ok(result);
    }
}

All the actual logic lives in the query handler, which is independently testable without spinning up an HTTP context.

Validation at the Boundary

I use FluentValidation hooked into the IRequestPreProcessor pipeline so validation fires before any handler runs — no guard clauses scattered through business logic.

public class CreateArticleCommandValidator : AbstractValidator<CreateArticleCommand>
{
    public CreateArticleCommandValidator()
    {
        RuleFor(x => x.Title).NotEmpty().MaximumLength(200);
        RuleFor(x => x.AuthorId).NotEmpty();
    }
}

The pipeline returns a consistent 400 Bad Request with field-level details before your handler ever sees bad data.

The Azure DevOps Pipeline

A two-stage pipeline: CI (build + test on every PR), CD (deploy to production on merge to main).

trigger:
  branches:
    include: [main]

pool:
  vmImage: 'ubuntu-latest'

stages:
  - stage: CI
    jobs:
      - job: BuildAndTest
        steps:
          - task: UseDotNet@2
            inputs:
              version: '8.x'

          - script: dotnet restore
            displayName: Restore

          - script: dotnet build --no-restore --configuration Release
            displayName: Build

          - script: dotnet test --no-build --configuration Release --logger trx
            displayName: Test

          - task: PublishTestResults@2
            inputs:
              testResultsFormat: VSTest
              testResultsFiles: '**/*.trx'

  - stage: CD
    dependsOn: CI
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: DeployProd
        environment: production
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureWebApp@1
                  inputs:
                    azureSubscription: '$(AzureServiceConnection)'
                    appName: '$(AppServiceName)'
                    package: '$(Pipeline.Workspace)/**/*.zip'

Key things that prevent the pipeline from becoming a problem:

  • Test results are always published — even on failure, so you see what broke without digging through logs.
  • CD is gated on CI — the dependsOn and condition ensure you can never deploy a failed build.
  • Environment approval gates — configure these in the Azure DevOps portal for your production environment if you want a manual sign-off step before release.

Health Checks

Add a /health endpoint so your load balancer and Azure Monitor can verify the API is actually responding:

builder.Services.AddHealthChecks()
    .AddDbContextCheck<AppDbContext>()
    .AddUrlGroup(new Uri("https://dependency.example.com"), "external-api");

app.MapHealthChecks("/health");

A 200 from /health means the database connection is alive and critical dependencies are reachable — not just that the process started.

What Makes This Hold Up

The combination of thin controllers, validated commands, and a pipeline that runs all tests on every PR means that by the time code reaches production, it's been exercised. The Azure App Service deployment with health check polling gives you a fast feedback loop if something is wrong after release.

The pipeline YAML lives in source control alongside the code, so infra changes go through code review like everything else. No surprise pipeline edits that break deployments weeks later.
